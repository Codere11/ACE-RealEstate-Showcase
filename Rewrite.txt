ACE-RealEstate Rewrite Plan
Option B: Spring Boot Main App + Angular Chatbot + Spring Admin Dashboard
Created: 2026-05-06

====================================================================
1. EXECUTIVE DECISION
====================================================================

We are proceeding with Option B.

That means:
- Spring Boot 21 becomes the main application and public entry point.
- The existing Angular visitor chatbot is kept for now.
- The manager/admin dashboard is rebuilt inside Spring Boot using Thymeleaf + HTMX.
- PostgreSQL remains the main database.
- Live help uses LiveKit Cloud instead of a local self-hosted LiveKit service.
- The current Python/FastAPI system may remain temporarily as an internal supporting service during migration, but the public product should increasingly look and behave like one coherent Java application.

Primary goals of this rewrite:
- Make the project much stronger as a Java showcase for Slovenian employers and clients.
- Simplify deployment and hosting.
- Present the product under one clean domain and one coherent route structure.
- Preserve existing value and working flows instead of throwing everything away.
- Reduce dev-only assumptions like localhost ports and Docker Compose-only networking.
- Support multi-tenancy cleanly and visibly.
- Prepare for client onboarding and production-style demos.

This is not just a technical rewrite.
It is a product packaging and credibility upgrade.


====================================================================
2. TARGET PUBLIC URL STRUCTURE
====================================================================

Public route structure:
- https://yourdomain.si/              -> main visitor chatbot / landing chatbot
- https://yourdomain.si/admin         -> manager login + dashboard
- https://yourdomain.si/demo          -> public demo tenant chatbot
- https://yourdomain.si/{tenantSlug}  -> tenant-specific chatbot

Example:
- https://yourdomain.si/nepremicnine
- https://yourdomain.si/dentist-demo
- https://yourdomain.si/fizioterapija

Why this structure is correct:
- The root domain becomes immediately usable for demos.
- Employers and clients do not need instructions or multiple links.
- /admin feels professional and product-like.
- /demo gives a safe public example tenant.
- /{tenantSlug} makes multi-tenancy visible and understandable.
- This structure is simple enough for a VPS deployment and strong enough for a SaaS story.

Reserved non-tenant routes that must never be interpreted as tenant slugs:
- admin
- api
- actuator
- assets
- static
- public
- pay
- health
- docs
- favicon.ico
- robots.txt
- sitemap.xml
- login
- logout
- error

Tenant resolution rule:
- /                  -> default tenant OR marketing/demo chatbot
- /demo              -> demo tenant
- /{tenantSlug}      -> tenant chatbot if slug exists and is active
- if slug does not exist -> show 404 or branded tenant-not-found page

3. FINAL ARCHITECTURE VISION
====================================================================

Target architecture:

A. Main application
- Java 21
- Spring Boot
- Serves public routes, admin UI, APIs, auth, tenant resolution, static chatbot assets, and most business logic.

B. Visitor chatbot
- Existing Angular app kept for now.
- Built into static assets.
- Served by Spring Boot.
- Uses relative paths for API communication.
- Reads tenant from the browser path.

C. Admin / manager dashboard
- Rebuilt in Spring Boot using Thymeleaf + HTMX.
- Routed under /admin.
- Handles authentication, tenant management, lead review, qualifier management, live help controls, payment settings, and operational workflows.

D. Database
- PostgreSQL.
- Proper schema evolution via Flyway or Liquibase.
- No production dependence on startup auto-create behavior.

E. Live help
- LiveKit Cloud.
- Spring Boot stores/configures sessions and generates visitor/manager access metadata or tokens.

F. AI / qualification logic
Two acceptable phases:
1. Transitional phase:
   - keep Python logic as an internal/private service if needed.
   - Spring Boot orchestrates and presents it publicly.
2. Final phase:
   - move important business orchestration into Java.

G. Payments
- Stripe integration stays.
- Spring Boot should own callback URLs, public payment URL generation, settings, and dashboard integration.

Important strategic principle:
The user should experience one product, not a pile of services.
Even if migration is incremental, the public shape of the system must look unified.


====================================================================
4. WHY OPTION B IS THE RIGHT CHOICE
====================================================================

Option B is the best compromise between speed, business value, and career value.

Reasons:
- Full rewrite right now would be slow and risky.
- Purely keeping the existing stack would weaken the Java showcase.
- Spring Boot as the main shell gives a much stronger architectural story.
- Keeping Angular for the chatbot preserves working UI and saves time.
- Thymeleaf + HTMX is a fast path to a functional admin dashboard inside Java.
- One-domain hosting becomes significantly simpler.
- It is easier to explain in interviews and client conversations.
- It aligns better with what many Slovenian companies respect: Java backend ownership, PostgreSQL, maintainable architecture, and practical product engineering.

What Option B optimizes for:
- showing Java capability
- preserving momentum
- faster public launch
- simpler deployment
- clearer multi-tenant story
- less infra friction than Railway-style multi-service hosting

5. NON-NEGOTIABLE PRINCIPLES FOR THE REWRITE
====================================================================

1. No hardcoded localhost URLs in production code.
2. Prefer relative URLs on the frontend whenever possible.
3. Public URLs must work on IP, staging domain, and production domain without code changes.
4. Backend-generated absolute URLs must be request-aware and env-overridable.
5. Public app must be accessible from one domain with clean paths.
6. Multi-tenancy must be explicit and understandable.
7. Secrets must come from environment variables, never hardcoded.
8. Database schema changes must use migrations.
9. Production behavior must not depend on Docker Compose-specific hostnames like backend:8000.
10. The admin dashboard must look intentional and product-like, not like an internal dev panel.
11. Demo tenant must be safe, polished, and easy to reset.
12. Every important feature shown to clients should have a deterministic demo path.

====================================================================
12. TENANT RESOLUTION DESIGN
====================================================================

Tenant resolution is central.

Rules:
- / -> default public experience
- /demo -> hardwired demo tenant slug, e.g. demo
- /{tenantSlug} -> tenant-specific experience if active
- reserved slugs should bypass tenant routing

Recommended logic:
1. Extract first path segment.
2. If no segment -> root/default route.
3. If segment is reserved -> route normally.
4. Else lookup tenant by slug.
5. If found and active -> render chatbot shell.
6. If not found -> show tenant-not-found page.

The Angular chatbot should then receive tenant context through one of these methods:
- server renders small bootstrap JSON into page
- meta tag with tenant slug
- data attribute on root element
- or Angular reads location path directly and asks backend for the tenant config

Preferred approach:
- Angular reads path and also asks backend for current tenant info.
- backend remains source of truth.

16. URL AND HOST AUTO-DETECTION STRATEGY
====================================================================

Yes, the app should work in a common-sense way on whatever host/IP it is deployed to.

This should be implemented as follows:

A. Frontend rule
- use relative URLs for all same-origin calls
- do not hardcode localhost or domains

Examples:
GOOD:
- /api/auth/login
- /api/organizations/current
- /chat
- /chat-events/poll
- /pay/abc123

BAD:
- http://localhost:8000/api/auth/login
- http://127.0.0.1:8000/chat
- https://api.something-hardcoded.com

B. Backend absolute URL rule
For things that must generate full URLs, use this priority:
1. explicit env override if set, e.g. PUBLIC_BASE_URL
2. forwarded headers from reverse proxy
3. request scheme + host + port

This allows the app to work:
- on raw VPS IP
- on staging domain
- on production domain

Use env overrides for integrations that really need stable public URLs:
- Stripe
- emails
- webhooks
- externally shared links

C. Forwarded headers
Spring Boot should trust and use forwarded headers correctly behind proxy/load balancer.
Examples to support:
- X-Forwarded-Proto
- X-Forwarded-Host
- X-Forwarded-Port

D. Final principle
Automatic by default, explicit when needed.
That is the right balance.

19. LIVE HELP STRATEGY
====================================================================

The local LiveKit setup used in current development is not the desired public production approach.

We should use LiveKit Cloud.

Responsibilities in the new architecture:
- Spring Boot stores live session metadata
- Spring Boot creates or orchestrates session state
- Spring Boot exposes manager/visitor access details
- client-side apps connect to LiveKit Cloud using config provided by backend

Needed env/config:
- LIVEKIT_WS_URL
- LIVEKIT_API_KEY
- LIVEKIT_API_SECRET

Demo strategy for live help:
- it should be possible to disable live help per tenant
- demo tenant can show live help as optional premium capability
- if live help is down, core chatbot should still function

Important:
Live help must not be allowed to make the entire public demo brittle.
It should be an additive feature, not a single point of embarrassment.

Mitigation:
